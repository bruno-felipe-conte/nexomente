Design Tokens — A Linguagem Atômica

Todo sistema de design robusto começa com tokens: variáveis centralizadas que carregam significado semântico antes mesmo de carregar valor visual. Ao modificar um token, o sistema inteiro responde com coerência.

🧬 Tokens de Superfície (Camadas de Profundidade)

O NexoMente usa um tema dark. A hierarquia de profundidade deve ser explícita: superfícies mais escuras = mais "fundas", mais claras = mais próximas do usuário.

/* Base → Raised → Card → Elevated → Overlay */ --surface-base: #0D0E14; /* fundo absoluto da página */ --surface-raised: #13141D; /* sidebar, painéis secundários */ --surface-card: #1A1C28; /* cards e containers */ --surface-elevated:#1F2133; /* inputs, botões inativos */ --surface-overlay: #262940; /* tooltips, dropdowns, modais */ --surface-border: #2A2D45; /* bordas sutis e divisores */
📐 Tokens de Raio de Borda
--radius-xs: 4px; /* badges, chips pequenos */ --radius-sm: 8px; /* botões inline, tags */ --radius-md: 12px; /* cards secundários, inputs */ --radius-lg: 18px; /* cards principais, painéis */ --radius-xl: 24px; /* modais, drawers */ --radius-full:9999px; /* pills, avatares, toggles */
⚠️ Regra de ouro: raios maiores = mais amigável e jovem. O NexoMente atende estudantes, então prefira --radius-lg e --radius-xl para elementos primários. Reserve bordas mais afiadas para elementos de dado denso.
🌫️ Tokens de Sombra
--shadow-sm: 0 1px 4px rgba(0,0,0,0.4); --shadow-md: 0 4px 16px rgba(0,0,0,0.45); --shadow-lg: 0 8px 32px rgba(0,0,0,0.55); --shadow-glow-violet: 0 0 20px rgba(124,109,250,0.3); --shadow-glow-blue: 0 0 16px rgba(77,158,255,0.25); --shadow-glow-teal: 0 0 16px rgba(45,212,191,0.2);

Tipografia — A Voz do Sistema

A tipografia atual do NexoMente usa fontes do sistema (Inter/system-ui), que são funcionais mas não criam identidade. Para um produto voltado ao aprendizado, a tipografia deve ser legível, hierarquicamente clara e memorável.

Atual — Problemático

  • Inter/system-ui em todos os níveis — nenhuma identidade
  • Peso uniforme, dificulta hierarquia visual rápida
  • Tamanho único em muitos painéis (13-14px, cansativo)
  • Espaçamento entre letras neutro, monótono

Proposto — Sistema de 3 Fontes

  • Syne: Display e títulos — personalidade forte, geométrica
  • DM Sans: Corpo e UI — legibilidade máxima em tamanho pequeno
  • JetBrains Mono: Dados, código, valores numéricos
  • Hierarquia clara com 7 tamanhos definidos
TokenFonteSizeWeightUso
--type-display-xlSyne40–54px800Nome do app, landing, headlines de seção hero
--type-display-lgSyne28–36px700Títulos de página (ex: "Notas", "Flashcards")
--type-display-mdSyne20–24px700Títulos de cards, seções
--type-body-lgDM Sans16–17px400Texto de conteúdo principal, notas
--type-body-mdDM Sans14–15px400UI geral, labels, descrições
--type-body-smDM Sans12–13px400Metadados, timestamps, labels secundários
--type-labelDM Sans10–11px600CAPS labels, seções de nav, cabeçalhos de tabela
--type-monoJetBrains Mono12–13px400–500XP, contadores, código, hashes
Regra crítica de legibilidade: Nunca use texto menor que 12px para qualquer informação que o usuário precise ler. Labels de navegação devem ter no mínimo 13px. O tamanho mínimo de target de toque para texto interativo é 44px de altura de área clicável (independente do tamanho da fonte).

Espaçamento — Grade e Ritmo

O app atual tem espaçamento inconsistente entre seções. Algumas áreas são apertadas, outras têm ar demais. Uma grade de espaçamento baseada em múltiplos de 4px resolve isso sistemicamente.

📏 Escala de Espaçamento (base 4px)

--space-28pxGap entre ícone e label, padding de badges
--space-312pxPadding interno de botões, gap de row items
--space-416pxPadding de nav items, gap de formulários
--space-520pxPadding de cards pequenos
--space-624pxPadding de cards médios, gap entre seções
--space-832pxSeparação entre blocos de conteúdo

🗂️ Layout de Página — Regras Gerais
  • Sidebar: largura fixa de 240px (atual tem 220px, muito estreita para labels longas)
  • Conteúdo principal: padding horizontal de 40–48px
  • Máximo de largura de conteúdo: 960–1100px (previne linhas longas demais)
  • Cabeçalho de página: título + subtítulo com margin-bottom: 32px antes do primeiro elemento de conteúdo
  • Separação entre cards/blocos principais: 20–24px
  • Gap de grid de 2 colunas: 20px

Motion & Animação — Ritmo e Intenção

Animações bem calibradas fazem o app sentir vivo. O segredo é proporcionalidade: micro-interações em 80–150ms, transições de layout em 200–300ms, entradas de página em 300–500ms com stagger.

Micro
duration: 80ms
ease: ease-out
hover states, focus rings
Fast
duration: 150ms
ease: ease-out
botões, toggles, badges
Normal
duration: 220ms
ease: cubic-bezier(.4,0,.2,1)
cards, dropdowns, modais
Slow
duration: 350ms
ease: cubic-bezier(.4,0,.2,1)
entradas de página, layouts
Spring
duration: 500ms
ease: cubic-bezier(.34,1.56,.64,1)
XP gain, achievements
✨ Animações por Contexto
  • Entrada de página: fade-in + translateY(8px) → translateY(0), stagger de 60ms entre elementos, total 300ms
  • Nav item hover: background slide-in da esquerda + color transition, 120ms ease-out
  • Botão primário hover: box-shadow glow cresce + leve scale(1.015), 150ms
  • Card hover: translateY(-2px) + shadow aumenta, border-color ilumina, 180ms
  • XP ganho: número count-up + partículas pequenas voando para cima, spring 500ms
  • Flashcard virar: rotateY(180deg) 3D flip, 400ms ease-in-out, backface-visibility hidden
  • Timer do Pomodoro: pulso sutil de 2s quando ativo (opacity 1 → 0.7 → 1)
  • Streaming de IA: cursor piscante estilo terminal durante geração
  • Nós do Grafo: fade-in com scale(0) → scale(1), stagger por distância do centro
  • Notificações/Toast: slide-in do canto inferior direito, 280ms spring; auto-dismiss com shrink-height 300ms
⚠️ Respeite prefers-reduced-motion: todas as animações devem ser desativadas ou substituídas por crossfades instantâneos quando o usuário configurar redução de movimento no SO. Isso é ergonomia e acessibilidade.

Navegação Global — Sidebar Redesenhada

A sidebar atual é funcional mas sem alma. Não comunica qual módulo o usuário está, não agrupa por categoria, e o indicador de ativo é apenas uma mudança de cor. Podemos fazer muito melhor.

Problemas Atuais

  • Todos os itens de nav têm o mesmo peso visual
  • Sem agrupamento lógico (ferramentas vs. conteúdo)
  • Logo "NexoMente" sem hierarquia — parece um item de menu
  • Status "IA Online" escondido no rodapé sem destaque
  • "Drive: Sincronizado" aparece/desaparece, inconsistente
  • Sem atalhos de teclado visíveis

Sidebar Proposta

  • Logo com tagline e versão — ancora identidade do app
  • Grupos: Aprender (Notas, Estudo, Flashcards) + Criar (Gerador, Poemas) + Explorar (Chat IA, Grafo, Stats)
  • Item ativo: fundo destacado + barra lateral esquerda colorida + ícone colorido
  • Ícones com cor única por módulo (conforme tabela de cores)
  • Status de IA como "pílula" de status permanente e animada
  • Atalhos Ctrl+número visíveis no hover de cada item

Inputs & Formulários — Ergonomia de Entrada

  • Altura mínima de input: 44px — lei de Fitts aplicada ao teclado e mouse
  • Border padrão: 1px solid var(--surface-border) · Focus: 2px solid var(--neon-violet) + box-shadow glow sutil
  • Placeholder deve ter contraste ≥ 3:1 mas nunca igual ao texto preenchido — use --text-lo
  • Labels SEMPRE visíveis acima do input (não apenas placeholder) — floating labels são aceitáveis mas complexos para implementar com acessibilidade correta
  • Campos de busca: ícone de lupa à esquerda, botão "×" para limpar aparece apenas quando há conteúdo
  • Área de texto (textarea) deve ter resize vertical habilitado com altura mínima de 120px
  • Inputs de número devem usar font-mono para evitar que dígitos saltem

Botões — Hierarquia de Ação

Um bom sistema de botões comunica hierarquia instantaneamente. O NexoMente deve ter no máximo 5 variantes, cada uma com papel claro.

VarianteVisualUsoRegra
Primary Ação Principal Ação mais importante da tela Máx. 1 por tela/seção
Secondary Ação Secundária Ações de suporte, alternativas Usado junto ao Primary
Ghost Cancelar Ações destrutivas suaves, cancelamento Nunca destrutivo irreversível
Danger Excluir Ações destrutivas irreversíveis Sempre com confirmação modal
Icon Only ⋯ Ações secundárias em contexto de card/row Sempre com tooltip (title attribute)
Regra de loading: Todo botão que dispara operação assíncrona (API, save, generate) deve mostrar estado de loading com spinner + texto "Carregando…" e ficar desabilitado durante a operação. Nunca deixe o usuário clicando no vazio sem feedback.

Cards & Containers — Anatomia e Hierarquia

🃏 Anatomia de um Card Padrão
┌─────────────────────────────────────────────┐ │ [Ícone cor] Título do Card [badge/pill] │ ← Header │ Subtítulo ou metadado em --text-mid │ ├─────────────────────────────────────────────┤ │ │ │ Conteúdo principal — texto, dados, UI │ ← Body │ │ ├─────────────────────────────────────────────┤ │ [Ação Secundária] [Ação Primária] │ ← Footer (opcional) └─────────────────────────────────────────────┘
  • Padding interno padrão: 24px (não menos que 16px em cards pequenos)
  • Header e Footer separados por divisor 1px solid var(--surface-border)
  • Cards interativos (clicáveis) devem ter cursor:pointer + hover elevado (translateY -2px + shadow)
  • Cards de conteúdo longo devem ter overflow:hidden com "ver mais" gradiente fade na borda inferior
10

Feedback & Status — O App Que Responde

  • Toast notifications: canto inferior direito, máx. 3 simultâneas, auto-dismiss em 4s. Verde=sucesso, âmbar=aviso, vermelho=erro, azul=info
  • Estados vazios: Nunca "Nenhum card encontrado" puro. Sempre: ícone grande + título motivador + botão de ação. Ex: "Sem flashcards ainda — Criar meu primeiro card →"
  • Loading states: Skeleton screens para listas e cards (retângulos com shimmer) — nunca apenas spinner centralizado
  • Erros de formulário: Inline, abaixo do campo, em vermelho com ícone, animados com shake sutil
  • Progresso de operação longa: Barra de progresso com porcentagem ou etapas nomeadas — nunca spinner infinito sem contexto
  • Confirmações destrutivas: Modal com título claro, descrição das consequências, botão cancelar à esquerda e confirmar à direita (vermelho)
11

Tela de Notas — O Coração do App

Problemas Atuais

  • Layout de 3 painéis (sidebar, lista, editor) em uma só tela — confuso em resolução menor
  • Bibliotecas como lista plana, sem context visual de quantidade/tipo
  • Toolbar do editor com 15+ ícones sem agrupamento lógico
  • Botões IA, Info, Tags, Editar no header — muito congestionado
  • Estado "IA offline" escondido no canto, sem destaque

Redesign Proposto

  • Layout 3 painéis responsivo: painéis colapsam em tablets, modal em mobile
  • Bibliotecas com cards visuais mostrando preview de cor e contagem de notas
  • Toolbar do editor agrupada: Formatação | Estrutura | Mídia | IA | Ações
  • Ações IA como floating button dentro do editor (não no header)
  • Status de IA como banner sutil no topo do painel quando offline
📒 Especificações do Editor de Notas
  • Modo de foco: botão "Tela cheia" expande o editor para 100% da viewport, esconde sidebar e painel lateral. Atalho: F11 ou Ctrl+Shift+Enter
  • Autosave: indicador sutil no footer do editor "Salvo há 2s" — nunca botão "Salvar" explícito para conteúdo de nota
  • Toolbar flutuante: ao selecionar texto, uma mini-toolbar contextual aparece próximo à seleção com: Bold, Italic, Link, Highlight, Ask AI — elimina necessidade de ir até a toolbar principal
  • Slash commands: digitar "/" no início de uma linha exibe menu de inserção (heading, lista, tabela, imagem, flashcard, etc.) — extremamente eficiente
  • Tags visuais: tags da nota aparecem como pills coloridas abaixo do título, não escondidas atrás de um botão
  • Breadcrumb: "NexoMente / Notas / Inbox / Nome da Nota" — sempre visível, clicável

Tela de Estudo — Timer & Sessões

⏱ Redesign do Timer Pomodoro
  • Timer visual: círculo SVG com progresso animado (stroke-dashoffset), não apenas números. Cores: Teal para estudo, Âmbar para pausa
  • Matéria selecionada: campo de seleção de matéria ANTES do iniciar — o timer não deve começar sem contexto de matéria. Prevenção de frustração
  • Ciclos visíveis: indicador dos ciclos Pomodoro como dots: ●●●○ (3 de 4 concluídos)
  • Notificação sonora: toggle de som no header do timer. Som suave de sino ao término
  • Tarefa atual: campo de texto "O que estou estudando agora?" abaixo do timer — ancora o foco
  • Progresso por matéria: barras de progresso com cores únicas, mostrando % do tempo-meta semanal atingido
  • Histórico da sessão: lista de sessões do dia com duração, matéria e horário — visível ao lado do timer em desktop

Flashcards — Revisão Espaçada Reimaginada

🃏 Gerenciamento de Cards
  • Cards como "deck" visual: ao invés de lista plana, mostrar cards empilhados em grupos/baralhos — a metáfora física ancora a memória
  • Filtros visíveis: "Todos | Hoje | Novos | Revisão | Dominados" como tabs com contagem por badge
  • Criação rápida: botão "+ Card" abre modal com 2 campos: Frente / Verso + campo de matéria/tag. Modal pequeno e focado, não nova página
  • Geração por IA: botão "Gerar da nota" no contexto de uma nota — cria flashcards automaticamente do conteúdo
🔄 Modo Revisão
  • Card grande, centralizado: ocupa 70% da viewport. Lado A em branco, virar com clique ou Espaço
  • Flip animation 3D: rotateY(180deg) com perspective, backface-visibility:hidden. Sensação física de virar um card
  • Avaliação visual: após ver o verso, 4 botões coloridos: 🔴 Errei (1) · 🟠 Difícil (2) · 🟢 Bom (3) · 💜 Fácil (4). Cores fixas para criar memória muscular
  • Progresso da sessão: barra no topo com X/total cards revisados, não apenas números
  • Celebração ao final: animação de confetti ou XP gain quando sessão completa. Motivação intrínseca

Gerador IA — Da Matéria-Prima ao Banco

Problemas Atuais

  • Dois blocos de upload lado a lado sem hierarquia clara
  • Área de texto sem indicador de tamanho/limite
  • "Processar Texto" como link visual — sem peso de CTA
  • Sem preview do que será gerado
  • Banco (0) como tab vazia sem estado de incentivo

Redesign Proposto

  • Flow em etapas numeradas: 1 Entrada → 2 Configurar → 3 Gerar → 4 Revisar
  • Upload com drag-and-drop visual animado (borda pulsante ao arrastar)
  • Configuração: nº de questões (slider), tipo (múltipla escolha/dissertativa/V/F), dificuldade
  • Preview das questões antes de salvar no banco, com opção de editar ou rejeitar
  • Banco com cards de questões filtráveis por matéria e tipo
⚙️ Flow de Geração em Etapas
1
Entrada de Conteúdo

Upload de arquivo (.txt, .md, .pdf) com drag-and-drop animado OU textarea com contador de caracteres. Detectar idioma automaticamente. Mostrar preview do texto extraído.

2
Configurar Geração

Quantidade (5/10/20 ou custom via slider), tipo de questão, dificuldade, matéria de destino. Estimativa de tempo de geração visível.

3
Gerar com IA

Botão CTA primário com âmbar. Durante geração: skeleton de questões aparecendo em tempo real (streaming), não aguardar tudo de uma vez.

4
Revisar e Salvar

Cada questão gerada aparece como card editável. Checkboxes para selecionar quais salvar. "Salvar Selecionadas no Banco" como ação final.

Chat IA — Assistente de Aprendizado

💬 Redesign da Interface de Chat
  • Contexto de nota visível: quando uma nota está selecionada, aparecer chip/pill compacto no topo do chat com o título da nota e botão de remover contexto
  • Modelo selecionado: dropdown de modelo mais visível, com nome descritivo do modelo e indicador de capacidade (velocidade vs. qualidade)
  • Prompt suggestions: estado vazio do chat deve ter 4–6 botões de sugestão de ação rápida: "Resumir minha nota", "Criar flashcards", "Explicar o conceito", "Fazer perguntas sobre o conteúdo"
  • Balões de mensagem: usuário à direita (roxo sutil), assistente à esquerda (card com borda sutil). Timestamps no hover
  • Ações em mensagens: hover em qualquer mensagem revela: Copiar | Regenerar | Criar Nota | Criar Flashcard
  • Streaming visual: cursor piscante + texto aparecendo palavra por palavra. Loading dots enquanto processa antes do streaming
  • Input multi-linha: textarea que expande automaticamente até 5 linhas, Shift+Enter para nova linha, Enter para enviar
  • Histórico de conversas: painel colapsável lateral com histórico de sessões anteriores com IA

Grafo de Conexões — Mapa do Conhecimento

🕸️ Experiência do Grafo
  • Estado vazio motivador: mostrar grafo de exemplo animado em ghost/transparência + call-to-action "Crie sua primeira nota para começar a mapear"
  • Nós coloridos por tipo: cada tipo de nota tem cor própria (Nota=azul, Livro=verde, Ideia=amarelo, etc.) — consistente com os chips de filtro já existentes
  • Hover em nó: card tooltip aparece com título, tipo, data e contagem de conexões
  • Click em nó: painel lateral desliza da direita com preview da nota + botão "Abrir nota"
  • Layout algorithm select: manter COSE mas adicionar opções Force-directed e Radial com preview instantâneo
  • Zoom controls: botões +/− e "Ajustar à tela" mais acessíveis — considerar scroll do mouse para zoom e arrastar para pan (já intuitivo mas precisa de onboarding visual)
  • Busca no grafo: campo de busca que filtra visualmente — nós não correspondentes ficam esmaecidos
  • Minimap: em grafos grandes, miniatura no canto inferior direito para navegação

Estatísticas — Gamificação com Substância

Problemas Atuais

  • XP Total como único número grande — sem contexto de nível
  • Todos os zeros não motivam — estado inicial desmotivador
  • "Notas por Tipo" e "Esse Mês" como seções vazias
  • Sem streak visual (sequência de dias)
  • Sem progressão de nível visível

Redesign Proposto

  • Card de nível com avatar/ícone, nome do nível, barra de XP até próximo nível
  • Estado inicial com metas sugeridas: "Complete sua primeira sessão de estudo"
  • Heatmap de calendário (estilo GitHub) para streak de dias
  • Gráfico de linha do tempo de XP ganho (sparkline)
  • Achievements/conquistas desbloqueáveis com ícones e descrições

Acessibilidade — Design Inclusivo

  • WCAG AA mínimo: contraste ≥ 4.5:1 para texto normal, 3:1 para texto grande. Verificar com ferramentas como Color Contrast Analyzer
  • Focus rings: visíveis e distintos — outline: 2px solid var(--neon-violet); outline-offset: 2px. Nunca outline: none sem substituto visível
  • Navegação por teclado: todo fluxo deve ser completável sem mouse. Ordem de tab lógica (esquerda → direita, cima → baixo)
  • ARIA labels: ícones sem texto devem ter aria-label. Botões com estados devem usar aria-pressed, aria-expanded
  • Alt text em imagens: toda imagem funcional precisa de alt descritivo. Imagens decorativas: alt=""
  • Tamanho de target: mínimo 44×44px para qualquer elemento interativo (WCAG 2.5.5)
  • Modo de alto contraste: usar CSS media @media (forced-colors: active) para ajustes
  • Não depender só de cor: estados de erro, sucesso, etc. devem ter ícone + texto além da cor

Princípios Gerais — A Filosofia do NexoMente

Missão do Design: O NexoMente deve ser o app de estudo que sente que está do lado do usuário. Cada interação deve reduzir fricção cognitiva, não adicionar. O app deve desaparecer — o que deve brilhar é o aprendizado.
🧭 Os 7 Princípios Inegociáveis
1
Clareza sobre inteligência

Nunca sacrifique clareza por sofisticação. Se o usuário precisa pensar sobre como usar a UI, a UI falhou. Cada tela deve ter uma pergunta clara: "O que quero que o usuário faça aqui?"

2
Contexto sempre visível

O usuário nunca deve se perguntar "onde estou?" ou "o que estou fazendo?". Breadcrumbs, títulos de seção, indicadores de módulo e status de sessão devem estar sempre presentes.

3
Zero estados vazios desmotivadores

Cada tela vazia é uma oportunidade de engajamento. "Nenhum flashcard" → "Que tal criar seu primeiro card? A revisão espaçada muda tudo." Estados vazios são marketing interno.

4
Feedback imediato

Toda ação deve ter uma resposta visual em menos de 100ms. Se a operação demora, mostrar progresso. Nunca deixar o usuário sem saber se o app recebeu o comando.

5
Progressividade da complexidade

O app deve ser simples para começar, poderoso para crescer. Funcionalidades avançadas (como slash commands, atalhos de teclado, configurações de SM-2) devem existir mas não intimidar novos usuários.

6
Consistência sistêmica

Um botão roxo = ação primária, sempre. Uma cor teal = módulo de Estudo, sempre. Uma vez que o usuário aprende a linguagem visual do app, ela deve nunca mudar arbitrariamente. Tokens de design aplicados rigorosamente.

7
Leveza como respeito ao usuário

Não sobrecarregue com notificações, tooltips desnecessários, modais de confirmação para ações reversíveis, ou popups de marketing. O usuário veio estudar. Respeite isso.
